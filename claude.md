CLAUDE.md — DevNavi 프로젝트 컨텍스트 (최종)
한 줄 요약

"Copilot이 짜주는 코드, 그게 뭔지는 알아야 하잖아."


디자인 철학 (코드 작성 시 항상 이걸 기준으로 판단)
왜 만드는가 (Pain Point)
AI(Copilot, Cursor, Claude, GPT 등)가 코드를 뚝딱 만들어주는 시대가 됐다.
편하긴 한데 문제가 생겼다:

작은 것도 직접 수정 못하는 개발자가 늘고 있다
"이게 어디 있지?" "이 명령어가 뭐였지?" 매번 구글링한다
AI 의존도만 높아지고 실력은 안 늘어난다
AI 없으면 아무것도 못하는 개발자가 된다

핵심 문제: AI가 대신 해주는 건 많은데, "내가 뭘 하고 있는지" 모르는 개발자가 늘고 있다.
우리의 포지셔닝
AI 에이전트 (Claude/GPT/Gemini) = 두뇌 🧠
  → 코드 설명, 힌트, 로드맵 생성 등 "생각하는 것"
  → 유저가 자기 API 키로 직접 연결. 우리가 AI를 제공하는 게 아님.

DevNavi = 효율적인 껍데기 + 토큰 절약기 💡
  → 그 두뇌를 "최소 비용으로, 최고의 UX로" 전달하는 것
  → AI를 만드는 게 아니라, AI를 더 잘 쓰게 해주는 도구
Copilot과의 관계

경쟁이 아니라 공존이다
Copilot/Cursor 등은 "코드를 대신 짜주는" 도구
DevNavi는 "그 코드가 뭔지 이해시켜주는" 도구
Copilot이 짜주면, DevNavi가 "이게 이런 거야" 알려주고, 작은 건 직접 수정할 수 있게 가이드
어떤 AI를 쓰든 상관없음 — DevNavi는 에디터에 있는 코드를 읽는 거니까

UX 철학

UX가 유일한 경쟁력이다 — AI는 유저 거 쓰는 거니까, 우리가 빛나야 하는 건 "어떻게 보여주느냐"
화려함보다 실용성 — 눈에 띄는 게 아니라, 자연스럽게 녹아드는 UX
네이티브 느낌 — VSCode에 원래 있던 기능처럼 느껴져야 함
최소한의 간섭 — 유저의 코딩 흐름을 방해하지 않음
성취감 설계 — 프로젝트 네비는 게임 퀘스트처럼. 하나씩 깨는 재미

토큰 절약 철학

절약은 조용히 일어나야 한다 — 화려한 대시보드가 아님
평소에는 안 보이고, 궁금할 때 한 번 열어보는 "건강검진 리포트" 느낌
핵심 가치: "같은 질문을 몇 번이나 반복했는지" 보여줘서 → 유저가 "아 이건 외워야겠다" 스스로 깨닫게 하는 것
이것도 결국 "대신 해주는 게 아니라, 할 수 있게 해주는 것"의 연장선

핵심 원칙 요약

"대신 해주는 게 아니라, 할 수 있게 해주는 것" — 답을 주지 않고 힌트를 준다
공존 — Copilot/Claude/GPT 어떤 AI와도 같이 쓸 수 있다
UX가 제품이다 — AI는 남의 것, 우리의 무기는 보여주는 방식
조용한 도움 — 눈에 안 띄게 작동하고, 필요할 때만 나타남
성장 유도 — 반복 질문 패턴을 보여줘서 스스로 깨닫게 함

UI 전략 — TreeView vs Webview
TreeView 사용 (가볍고 네이티브한 UI)

치트시트: 카테고리 펼치기/접기, 명령어 목록, 클릭 복사
프로젝트 네비: 부트캠프식 체크박스 목록, Phase별 진행
VSCode 탐색기처럼 자연스러운 느낌, 무겁지 않고 빠름

Webview 사용 (복잡한 UI가 필요할 때만)

코드 가이드: 인라인 설명, 하이라이트 UI
React + Tailwind로 구현
Webview를 남용하지 않는다 — 꼭 필요할 때만 사용


핵심 기능 4가지
기능 1. 스마트 치트시트 (TreeView)

git, 터미널, npm, 단축키 등 자주 쓰는 명령어 카테고리별 펼쳐보기
"이게 뭐야 / 언제 써 / 예시" 3줄 설명
클릭 한 번에 터미널 복사 + 검색
데이터는 로컬 JSON (API 호출 없음 = 토큰 0원)
커스텀 명령어: 유저가 자기만의 명령어를 추가/수정/삭제 가능

기본 제공 명령어 (cheatsheet.json) + 유저 커스텀 명령어 (VSCode globalState에 저장)
"내가 자주 쓰는 명령어" 카테고리 자동 생성
유저가 카테고리명, 명령어, 설명을 직접 입력 가능


UX: IDE 사이드바에서 바로 펼쳐짐. 창 전환 없이 0.5초 안에 찾고 복사. VSCode 탐색기와 같은 수준의 자연스러움.

기능 2. 프로젝트 네비게이터 — 부트캠프식 가이드 (TreeView + AI)

핵심 컨셉: 부트캠프 멘토가 IDE 안에 살고 있는 느낌
"어떤 프로젝트 만들 거야?" 또는 기획서 첨부 → AI가 Phase별 체크리스트 생성
TreeView에 체크박스 목록으로 표시:

  📁 할일 관리 앱
    ✅ Phase 1: 프로젝트 세팅
      ✅ create-next-app 실행
      ✅ 폴더 구조 잡기
      ✅ 타입스크립트 설정
    🔲 Phase 2: 기본 UI 만들기
      🔲 할일 입력 컴포넌트
      🔲 할일 목록 컴포넌트
      🔲 완료/삭제 버튼
    🔲 Phase 3: 상태 관리
      🔲 useState로 할일 CRUD
      🔲 로컬스토리지 저장
    🔲 Phase 4: 스타일링
      ...

각 항목 클릭 → "이거 어떻게 해?" → AI가 힌트만 줌 (답을 통째로 안 줌)
체크하면 다음 단계 자동 열림 — 성취감 극대화
Phase 전체 완료 시 축하 메시지
진행률 상태바에 표시 "Phase 2/4 · 5/12 완료"
프로젝트 로드맵은 VSCode globalState에 저장 (다시 열어도 진행 상황 유지)
UX: 게임 퀘스트처럼 하나씩 깨나가는 느낌. 막막함을 없애주는 것이 목적.

기능 3. 코드 가이드 — AI (Webview)

코드 블록 선택 → "이거 뭐하는 코드야?" → 쉬운 말로 설명
답을 안 주고 힌트만 줌 — "이 부분 직접 수정해볼래?"
Copilot이 짜준 코드를 이해할 수 있게 도와줌
에러 났을 때 정답을 안 알려주고 "여기가 문제야. 이렇게 생각해봐" 방향만 제시
UX: 코드 옆에 인라인으로 설명. 해당 라인 하이라이트. 별도 창이 아니라 코드 맥락 안에서.

기능 4. 토큰 절약 시스템 (조용히 작동)
중요: 이 기능은 화려하게 눈에 띄면 안 됨. 백그라운드에서 조용히 작동.
자동으로 작동 (유저가 신경 안 써도 됨):

로컬 우선 응답: 치트시트 질문은 JSON에서 바로 답 (API 호출 0원)
스마트 캐싱: 같은/비슷한 질문 → 캐시된 답변 재사용
프롬프트 압축: 불필요한 코드 컨텍스트 자동 제거

유저가 보고 싶을 때만 보는 것들:

상태바에 작은 아이콘 하나: "🪙 3회 절약" 정도만 (안 눌러도 됨)
클릭하면 상세 패널:

오늘/이번 주 토큰 사용량
캐시로 절약한 횟수
내가 자주 반복하는 질문 TOP 5 ← 핵심 기능

"git stash 뭐야?" → 3번 물어봄 → "치트시트에 추가할까요?"
같은 걸 자꾸 물어보는 패턴 → 유저가 "아 이건 외워야겠다" 스스로 깨달음


모델별 비용 비교 ("GPT-4o로 하면 ₩80, Haiku로 하면 ₩5")


UX 원칙: "건강검진 리포트" 느낌. 평소에는 안 보이고, 궁금할 때 한 번 열어봄.


안 하는 것 (이것도 중요)

코드 자동완성 안 함 → Copilot 영역
복잡한 아키텍처 설계 안 함 → 너무 딥하지 않게
코딩 테스트 출제 안 함 → 교육 플랫폼이 아님
완성 코드 제공 안 함 → 답을 주면 의미 없으니까
토큰 대시보드를 화려하게 만들지 않음 → 조용히 작동


배포 전략

전체 무료 배포 (포트폴리오 + 유저 확보 목적)
LLM 비용 0원 (유저 본인 API 키 사용)
수익 모델(Free/Pro 구독)은 설계 완료 상태 — 추후 사업자 등록 후 적용 가능
GitHub에 오픈소스로 전체 공개

기술 스택

Extension: VSCode Extension API (TreeView + Webview), TypeScript, React + Tailwind (Webview용)
AI: 유저 API 키 기반 (OpenAI/Claude/Gemini), 프롬프트 엔지니어링, 로컬 캐싱
Web: Next.js 14+, Supabase (DB/Auth), Vercel
배포: VSCode Marketplace + Vercel

개발 로드맵

1~2주차: 치트시트 TreeView 완성 (기본 명령어 + 커스텀 + 검색 + 복사)
3주차: 프로젝트 네비 TreeView (부트캠프식 체크리스트 + AI 연동)
4주차: 코드 가이드 Webview + 토큰 절약 시스템 + 웹사이트
5주차: 배포 + README + 데모 영상 + 마케팅

코드 품질 원칙 (매우 중요)

최적화 우선: 모든 코드는 성능과 효율성을 최우선으로 작성한다
보일러플레이트 활용: 검증된 패턴과 보일러플레이트를 적극 활용하여 바퀴를 재발명하지 않는다
불필요한 코드 금지: 쓰지 않는 import, 빈 함수, 주석처리된 코드 등 죽은 코드를 남기지 않는다
DRY 원칙: 같은 로직을 반복하지 않고 공통 유틸로 추출한다
최소한의 의존성: 꼭 필요한 패키지만 설치. 라이브러리 하나 때문에 번들 사이즈 늘리지 않는다
타입 안전성: TypeScript를 쓰는 이유가 있으니 any를 남발하지 않는다. 제대로 된 타입 정의.
번들 최적화: webpack tree-shaking이 잘 되도록 코드 구조를 잡는다
메모리 관리: 이벤트 리스너, 구독 등은 반드시 dispose/cleanup 한다. Extension 생명주기 관리 철저.
로딩 속도: Extension 활성화 시 무거운 작업 하지 않는다. 필요할 때 lazy loading.
캐싱 전략: 같은 데이터를 반복 읽지 않는다. JSON은 한 번 읽고 메모리에 올려둔다.

주의사항

코드 작성 시 한국어 주석 사용
치트시트 + 프로젝트 네비는 TreeView로 구현 (Webview 아님)
코드가이드만 Webview (React + Tailwind)
토큰 절약 시스템은 눈에 안 띄게 — 상태바 작은 아이콘 + 클릭 시 상세
치트시트 기본 데이터는 src/data/ 폴더의 JSON 파일 (DB 아님)
커스텀 명령어 + 프로젝트 진행 상황 + 토큰 기록은 VSCode globalState에 저장
유저 API 키는 VSCode 내장 SecretStorage에 저장
UI는 화려함보다 실용성. VSCode에 원래 있던 기능처럼 자연스러워야 함.
